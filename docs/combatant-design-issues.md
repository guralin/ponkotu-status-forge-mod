# 戦闘周辺コードの調査まとめ（現状）

DDD（エリック・エヴァンス）観点で境界と一貫性を確認し、テスト観点は「仕様の固定」を重視しています。ここでは現状だけを簡潔に整理します。

## いまの構造（要約）
- Foundry 依存の変換は `repository` 配下に集約され、ドメインは Foundry に直接依存しません。
- ステータス定義は `statusDefinitions` が唯一の定義元で、属性キーも `StatusDefinition.attribute` に統合されています。
- 状態は `StatusSet` を中心に管理され、ターン処理やダメージ処理はそこを参照します。
- HP/Barrier/Constitution/SAN の更新は `Combatant` の操作メソッド経由に寄せています。

## 現在の課題（重要度順）
### A. Combatant の直接代入が残る（中）
- 更新メソッドは導入済みだが、フィールドが public のため直接代入が可能です。
- 将来的に不変条件が崩れるリスクが残ります。

### B. loadByActorId の戻り値が文脈不明（中） → 解決済み
- `combatId` / `combatantId` を削除し、Actor 変換用レコードに整理しました。
- 「戦闘外の Actor 変換」と「戦闘中の Combatant 取得」は、必要になった時点で別メソッド/別型として切り出す前提です。

### C. pending の処理順は仕様依存（低）
- pending を stack に移してから `onTurnStart` を呼びます。
- pending を参照したい効果がある場合、仕様とのズレが起きます。

## 次の改善候補（短く）
- **不変条件の強制**: フィールドを private 化し、更新はドメインメソッドに限定。
- **型で文脈分離**: 戦闘外/戦闘内の取得結果を型で区別。
- **仕様の固定**: pending の順序をテストで明文化。
